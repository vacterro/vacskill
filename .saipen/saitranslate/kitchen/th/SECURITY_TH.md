# นโยบายความปลอดภัย

## ขอบเขต

SAIPEN เป็นข้อกำหนดบวกกับชุดสคริปต์สำหรับการติดตั้ง/ส่งออกในเครื่องเพียงเล็กน้อย
(`bootstrap/inject.ps1`/`.sh`, `uninstall.ps1`/`.sh`, `export.ps1`/`.sh`).
มันไม่ได้รันเซิร์ฟเวอร์, ไม่ได้รวบรวมข้อมูล telemetry (telemetry),
และไม่มีการส่งต่อข้อมูลใดๆ ไปยังที่ใด. ทุกอย่างที่สคริปต์เหล่านี้
ดำเนินการ คือการเขียนข้อมูลลงในระบบไฟล์ภายในเครื่องที่คุณควบคุม
อยู่แล้ว (ไม่ว่าจะเป็น `~/.claude`, `~/.gemini`, หรือ `.saipen/` ภายใน
โปรเจกต์ ฯลฯ) ซึ่งแต่ละไฟล์จะได้รับการป้องกันโดยการสำรองข้อมูล
เป็นไฟล์ `.bak` โดยอัตโนมัติก่อนที่จะมีการเปลี่ยนแปลงครั้งแรก

สิ่งที่คุ้มค่าแก่การรายงานเกี่ยวกับความปลอดภัยมีเพียง 2 อย่าง:
1. สคริปต์ bootstrap ที่เข้าไปทำการแก้ไขกับระบบไฟล์ หรือ ประวัติ
   git ของคุณ เกินกว่าขอบเขตที่ตัวโค้ดคอมเมนต์/README อธิบายเอาไว้
2. ข้อควรระวังตามโปรโตคอล (RFC.md ข้อ 1.1 -- ห้ามนำคีย์ API,
   โทเคนการเข้าถึง, รหัสผ่าน เข้าไปเก็บใน `STATE.md`/`BOARD.md`/
   `LOG.md`/`KNOWLEDGE/`/`kitchen/` เด็ดขาด) เกิดช่องโหว่จริงๆ ที่ทำให้
   agent ที่ใช้งานระบบ SAIPEN สามารถหลุดข้อมูลลับเข้าไปอยู่ใน
   ไฟล์ที่มีการคอมมิต (commit) ได้

## เวอร์ชันที่รองรับ

จะรองรับเฉพาะรีลีสที่ผูกแท็ก (tagged release) ล่าสุดบนกิ่ง (branch)
`main` เท่านั้น. นี่คือข้อกำหนดของโปรโตคอล (protocol specification) ไม่ใช่
บริการระยะยาว (long-lived service) -- ดังนั้นไม่มี LTS branch

## การรายงานช่องโหว่

เปิดประเด็น (Issue) ในหน้า GitHub. ถ้ารายงานนั้นเกี่ยวกับปัญหาที่
มีความเสี่ยงอย่างแท้จริง หรือมีผลกระทบในปัจจุบัน (ไม่ใช่เรื่อง
สมมติขึ้น), ให้คุณตั้งค่าเรื่องนั้นเป็นเอกสารข้อแนะนำแบบส่วนตัว/ด้านความปลอดภัย (private/security
advisory) ผ่านแท็บ **Security** ภายในที่จัดเก็บข้อมูลนี้ ("Report a vulnerability")
แทนที่จะเปิดเป็น Issue แบบสาธารณะ, เพื่อป้องกันไม่ให้ข้อมูลนั้น
มองเห็นได้ต่อสาธารณะก่อนที่จะออกตัวแก้ไข (fix)

โปรดระบุ: สคริปต์ ตัวแปร หรือข้อกำหนดใน RFC ข้อใด, เหตุการณ์สมมติ
แบบเป็นรูปธรรม (concrete scenario), สิ่งที่เกิดขึ้นจริง (what actually
happens) เทียบกับสิ่งที่ควรเกิดขึ้น (what should happen).
โดยใช้มาตรฐานเดียวกันกับเมื่อมีการรายงานข้อบกพร่องอื่นๆ (bug report)
(อ่านเพิ่มเติมใน `CONTRIBUTING.md`)
