# ข้อกำหนด SAIPEN (SAIPEN Specification)

## บทคัดย่อ
**เป้าหมายการออกแบบ #1: Agent ที่ถูกเริ่มต้นใหม่ (cold) และไม่มีประวัติการแชทใด ๆ จะต้องสามารถรันคำสั่ง `/saipen continue` และกลับไปทำงานได้อย่างมีประสิทธิผลภายในหนึ่งนาที โดยไม่ต้องขอให้ผู้ใช้เล่าบริบทซ้ำ**

SAIPEN รับประกันว่า AI agent ที่เข้ากันได้ตัวใดก็ตาม สามารถทำงานต่อในโปรเจกต์ใด ๆ ได้อย่างปลอดภัยโดยไม่ต้องสรุปงานใหม่ มันคือ ABI (Application Binary Interface) สำหรับ engineering AI agents—เลเยอร์ความเข้ากันได้ที่แก้ปัญหาอาการความจำเสื่อม ไม่ว่าวันนี้คุณจะใช้ Claude, พรุ่งนี้ใช้ Gemini, หรือมะรืนใช้ GPT ทุกตัวจะทำงานอิงตามสถานะโปรเจกต์เดียวกันโดยไม่ต้องให้คุณเล่าบริบทซ้ำ

### ปรัชญาหลัก: สถานะโปรเจกต์ > ความจำของโมเดล
ความจำควรอยู่คู่กับโค้ด ไม่ใช่ในหัวของโมเดลอื่น. SAIPEN เปลี่ยนกระบวนทัศน์จาก `โปรเจกต์ -> ความจำ -> LLM` ไปเป็น `โปรเจกต์ -> สถานะของ SAIPEN -> LLM`. ความจำนั้นเป็นของโปรเจกต์.

โดยหลักการพื้นฐาน SAIPEN ใช้โปรโตคอลสืบทอดแบบพกพาที่สำรองข้อมูลด้วยไฟล์ สำหรับ LLM agents. รูปแบบการปรับใช้อาจแตกต่างกันไป (Implementations MAY vary). แต่สัญญาระดับดิสก์ต้องมีความเสถียร (The on-disk contract MUST remain stable). ทุกสิ่งในโปรโตคอลนี้มีอยู่เพื่อรับใช้แบบทดสอบการสืบทอด (Continuation Test).

SAIPEN เน้นการวิวัฒนาการ ไม่ใช่ความคิดสร้างสรรค์. จุดประสงค์ของมันคือการทำซอฟต์แวร์ให้สมบูรณ์ ไม่ใช่การประดิษฐ์ขึ้นใหม่. ADD ขยายต่อยอดจากรูปแบบการออกแบบที่มีอยู่แล้ว, ธรรมเนียมปฏิบัติในอุตสาหกรรม, และฟีเจอร์ที่เห็นได้ชัดว่าควรมี.

- **`STATE`**: มีไว้เพื่อตอบคำถาม *"ฉันต้องทำอะไรตอนนี้?"*
- **`BOARD`**: มีไว้เพื่อตอบคำถาม *"ฉันกำลังหยิบงานไหนมาทำ?"*
- **`LOG`**: มีไว้เพื่อตอบคำถาม *"ทำไมเราถึงมาถึงจุดนี้ได้?"*
- **`KNOWLEDGE`**: มีไว้เพื่อตอบคำถาม *"ความจริงอันยั่งยืนของโปรเจกต์นี้คืออะไร?"*
- **`next_action`**: หัวใจของ SAIPEN. มันตอบคำถามที่ว่า *"ฉันต้องรันคำสั่งใดในวินาทีนี้เพื่อทำงานต่อ?"*

## บททดสอบ Litmus ของ SAIPEN

การเปลี่ยนแปลงหรือไอเดียใหม่ที่ถูกเสนอสำหรับโปรโตคอลนี้จะต้องผ่าน 3 คำถามต่อไปนี้:
1. มันทำให้การเปลี่ยนผ่านระหว่าง agents มีความน่าเชื่อถือมากขึ้นหรือไม่?
2. มันทำให้พฤติกรรมของโมเดลต่าง ๆ เป็นอันหนึ่งอันเดียวกันมากขึ้นหรือไม่?
3. มันช่วยลดความน่าจะเป็นในการสูญเสียบริบทหรือไม่?

หากคำตอบคือ "ไม่" ในอย่างน้อยสองข้อ ไอเดียนั้นจะถูกปฏิเสธ. SAIPEN ให้ความสำคัญกับความมีระเบียบวินัย, การทำซ้ำได้ (reproducibility), และความน่าเชื่อถือ เหนือความแปลกใหม่.

## สถาปัตยกรรม

โปรโตคอลมีลักษณะเชิงบรรทัดฐานอย่างเคร่งครัด (strictly normative). ในเชิงแนวคิด SAIPEN ถูกแบ่งออกเป็นสองเลเยอร์: **Core** (หลัก) และ **Maintenance** (บำรุงรักษา).
- **เลเยอร์ Core** รับประกันการทำงานต่อที่ปลอดภัยและเป็นกลางต่อผู้ให้บริการ.
- **เลเยอร์ Maintenance** เป็นโมเดลวิวัฒนาการซอฟต์แวร์อัตโนมัติที่สร้างอยู่บน Core.

ภายใต้สองเลเยอร์นี้ SAIPEN ได้แยกส่วนข้อกังวลที่ไม่มีวันพันกันออกเป็นสามส่วน:
**ความถูกต้องและการสืบทอดการทำงาน** (Core -- `STATE`/`BOARD`/`LOG`/`KNOWLEDGE`, การเจรจาขีดความสามารถ, การตั้งเช็คพอยต์), **วิวัฒนาการโดยปราศจากการดูแล** (Maintenance -- `HUNT`/`ADD`/`CLEAN`, ใช้งานได้อย่างสมบูรณ์แบบภายใต้คำสั่งเริ่มต้น `saipen`/`saipen continue`), และ **ปริมาณงานที่ทำได้ (throughput)** (โหมด Goal, Subagents -- ทั้งสองอย่างเป็นการเลือกเข้าอย่างชัดเจน (opt-in), §1.3/§2.4). ปิดใช้งานโหมด Goal: โปรโตคอลยังคงเหมือนเดิม ทำงานทีละหนึ่งทิคเก็ต. ปิดใช้งาน Subagents: `HUNT` จะรันทั้งหกหมวดหมู่ตามลำดับ ได้ผลลัพธ์เหมือนเดิม. ใช้ Core เดี่ยว ๆ โดยไม่มีเลเยอร์ Maintenance เลย: มันก็ยังคงใช้งานได้ -- cold agent ก็ยังคงสามารถทำงานต่อได้อย่างถูกต้อง. แต่ละเลเยอร์สร้างขึ้นบนเลเยอร์ที่อยู่ข้างใต้โดยไม่มีการย้อนศร; สิ่งที่อยู่ต้นน้ำ (upstream) จะไม่พึ่งพาสิ่งที่อยู่ปลายน้ำ (downstream).


## การเจรจาขีดความสามารถแบบสองทาง (Two-Way Capability Negotiation)
Agents จะไม่ประกาศออกมาเฉย ๆ ว่าสามารถทำอะไรได้บ้าง; โปรโตคอลจะเรียกร้องสิ่งที่จำเป็นขึ้นมา.
โปรเจกต์จะระบุ `requires: [filesystem, git, shell, python]` ไว้ในสถานะของมัน. จากนั้น agent จะจับคู่ตรวจสอบขีดความสามารถของโฮสต์เทียบกับความต้องการ แล้วล็อคตัวเองเข้าสู่โหมด (`mode`) (เช่น `full`, `read-only`).

## การบันทึกเหตุการณ์ด้วยกราฟ (Graph-Based Event Logging)
Log ใน SAIPEN ไม่ใช่สายอักขระเชิงเส้น (linear strings). มันประกอบกันเป็นกราฟแบบไม่มีวงวน (acyclic graph) ของการตัดสินใจ โดยใช้รหัสเหตุการณ์ Event IDs (`E-001`). รูปแบบนี้ช่วยให้สามารถทำการแยกแขนง การรวม agent และทำให้เส้นทางการตรวจสอบมีความแม่นยำสูง.

## บันทึกการตัดสินใจทางสถาปัตยกรรม (Architecture Decision Records - ADR)
Log เหตุการณ์ชั่วคราว ไม่ได้เป็นแหล่งรวบรวมความรู้อย่างถาวร. SAIPEN บังคับว่าการตัดสินใจทางสถาปัตยกรรมเชิงโครงสร้างจะต้องถูกบันทึกไว้อย่างถาวรในรูปแบบ ADRs (เช่น `KNOWLEDGE/ADR-001-use-sqlite.md`).

## ขอบเขตความพร้อมเพรียงและการกระจายตัว (Concurrency & Distribution Boundaries)
SAIPEN รับประกันความสมบูรณ์ของสถานะผ่านการอ้างสิทธิ์ (claim) ที่อิงตามไฟล์ (`owner`, `claim_time`) และกราฟเชิงลำดับ (`LOG.md`). อย่างไรก็ตาม **SAIPEN เป็นโปรโตคอลระบุสถานะ ไม่ใช่อัลกอริทึมมติเอกฉันท์แบบกระจายศูนย์ (distributed consensus algorithm).**
- **ระบบไฟล์แบบ Local/Shared**: การแก้ไขความขัดแย้ง จะขึ้นอยู่กับการเขียนระบบไฟล์แบบแบ่งแยกย่อยระดับปรมาณู ("คน commit ก่อนชนะ").
- **สภาพแวดล้อมแบบ Networked/Distributed**: หาก agents ปฏิบัติงานข้ามเครื่องจักรที่ไม่ได้เชื่อมต่อกัน และไม่ซิงค์ไฟล์แบบเรียลไทม์ การเกิด race condition เกี่ยวกับการอ้างสิทธิ์ใน `BOARD.md` ย่อมเกิดขึ้น. ในระบบที่กระจายศูนย์ระดับสูง สัญญาระดับดิสก์ของโปรโตคอล SAIPEN ต้องรักษาความคงที่เอาไว้ -- แม้ตัวสถานะของโปรเจกต์เองจะถูกเปลี่ยนแปลงตลอดเวลาผ่านกฎของ SAIPEN (§ 1.5 การตั้งเช็คพอยต์) แต่รูปทรงของโปรโตคอลที่กฎเหล่านั้นดำเนินตามจะไม่ถูกเปลี่ยนแปลงเลย.
