# การมีส่วนร่วมใน SAIPEN

SAIPEN เป็นข้อกำหนดมาก่อนที่จะเป็นการนำไปปฏิบัติ การมีส่วนร่วม
ส่วนใหญ่คือการเปลี่ยนแปลง `saipen/RFC.md`, โฟลเดอร์ `phases/*.md`
หรือเครื่องมือความสอดคล้อง (conformance tooling) ใน `tests/` --
ไม่ใช่ในตัวแอปพลิเคชันโค้ด

## ก่อนที่จะเสนอการเปลี่ยนแปลง

รัน [SAIPEN Litmus Test](SPEC_TH.md#the-saipen-litmus-test) กับ
ไอเดียของคุณ:
1. มันทำให้การเปลี่ยนผ่านระหว่าง agents มีความน่าเชื่อถือมากขึ้นหรือไม่?
2. มันทำให้พฤติกรรมของโมเดลต่าง ๆ เป็นอันหนึ่งอันเดียวกันมากขึ้นหรือไม่?
3. มันช่วยลดความน่าจะเป็นในการสูญเสียบริบทหรือไม่?

หากคำตอบคือ "ไม่" ในอย่างน้อยสองข้อ ถือว่าไอเดียของคุณอยู่นอกเหนือ
ขอบเขตของโปรโตคอลนี้ แม้ว่ามันอาจจะมีประโยชน์เมื่อใช้งานกับที่อื่นก็ตาม

## การรายงานข้อบกพร่อง (Gap)

เปิด Issue โดยระบุ:
- ไฟล์/ส่วนที่มีข้อบกพร่องอยู่ (RFC.md, phase doc, schema, หรือ test)
- หลักฐานที่เป็นรูปธรรม (คำพูดอ้างอิง, คำสั่งและผลลัพธ์ที่ได้, สถานการณ์
  ที่พฤติกรรมในปัจจุบันเกิดข้อผิดพลาด)
- สิ่งที่คุณคาดหวังให้เกิดขึ้นแทน

รายงานที่คลุมเครือ ("รู้สึกเหมือนว่าบางอย่างไม่ถูกต้อง") ย่อมดำเนินการ
แก้ไขได้ยากกว่ารายงานที่มีความเฉพาะเจาะจง ไม่ว่าจะเป็น `grep`/ตัวอย่าง
การทำให้เกิดข้อผิดพลาด. ดู template ของ Issue รายงานบั๊กสำหรับ
รูปแบบที่ควรใช้

## การแก้ไข/เพิ่มเติมเนื้อหา

1. อ่าน `saipen/RFC.md` และเอกสาร `phases/*.md` ที่เกี่ยวข้องทั้งหมด
   อย่างละเอียดก่อนที่จะแก้ไข -- ช่องโหว่ส่วนใหญ่ที่คุณคิดมักจะได้รับการ
   จัดการในที่อื่นแล้ว หรือจงใจจำกัดขอบเขตไว้ด้วยเหตุผลที่มีการบันทึกไว้แล้ว
2. ตรวจสอบ `CHANGELOG.md` และ `.saipen/KNOWLEDGE/decisions.md` สำหรับ
   โครงสร้างที่มีอยู่ก่อนหน้า อย่ารื้อฟื้นการตัดสินใจที่เคยเกิดขึ้นและถูก
   ปฏิเสธไปแล้วอย่างเงียบ ๆ -- หากคุณมีหลักฐานใหม่ว่าการปฏิเสธในอดีต
   เป็นสิ่งที่ผิด โปรดระบุอย่างชัดเจนในคำอธิบายของ PR (Pull Request)
3. การเปลี่ยนแปลงเชิงบรรทัดฐานทุกรายการ (MUST/MUST NOT/SHOULD) จำเป็น
   ต้องมีรายการใน `CHANGELOG.md` และ หากทำได้ ควรครอบคลุมในการ
   ทดสอบของ `tests/validate.sh` + `tests/validate.ps1` (ทั้งสองแพลตฟอร์ม)
   หรือไฟล์จำลองข้อมูลตัวอย่าง (fixture) ภายใต้ `tests/scenarios/`
4. รัน validator ทั้งสองตัวก่อนเปิด PR:
   ```bash
   bash tests/validate.sh
   powershell -File tests/validate.ps1
   ```
5. ปรับเวอร์ชัน (`VERSION`) ตามแผนใน `phases/ship.md` (ปรับตัวเลข patch
   สำหรับคำอธิบายในเอกสาร, ตัวเลข minor สำหรับพฤติกรรมเชิงบรรทัดฐาน
   ใหม่, และตัวเลข major สำหรับการเปลี่ยนแปลงข้อตกลงที่กระทบระบบเดิม)
   และต้องมั่นใจว่าเวอร์ชันใน badge ของ `README.md` นั้นตรงกัน -- โดยปกติ
   `tests/validate.sh`/`.ps1` จะตรวจสอบส่วนนี้อัตโนมัติเมื่อสั่งรันจากที่โคลนมา
   ใน repo นี้

## สไตล์

- สำหรับเอกสารเกี่ยวกับโปรโตคอล (Protocol) และขั้นตอนการทำงาน (Phase): สั้นกระชับ, ใช้คีย์เวิร์ด
  RFC-2119 เมื่อจำเป็น, ไม่มีน้ำ (เนื้อหาที่ไม่จำเป็น). ดู `saipen/STYLE.md`
- ทุกสิ่งในไฟล์นี้, ข้อความในคอมมิต (commit messages), ความคิดเห็นในโค้ด,
  และ CHANGELOG: แบบเรียบง่ายและเป็นมืออาชีพ. เสียงในโปรเจกต์แชท/LOG
  (`saipen/STYLE.md`) ไม่นำมาปรับใช้กับวัตถุที่ส่งมอบ (artifacts)

## สิ่งที่อยู่นอกเหนือขอบเขต

- การเปลี่ยน SAIPEN ให้เป็นระบบมติเอกฉันท์แบบกระจายศูนย์ (distributed
  consensus system). อ้างอิงหัวข้อ "Concurrency & Distribution Boundaries"
  ใน `SPEC.md`
- โครงสร้างตัวสร้างสัญญาณ (marker grammar) สำหรับ machine-parseable
  ใน LOG นอกเหนือจากโครงสร้างเดิมที่มีอยู่ `LOG.md` ยังคงเป็นบทความรอบ ๆ
  รูปทรงที่ถูกกำหนดไว้
- คำสั่ง `saipen doctor` หรือสิ่งที่คล้ายคลึงกันที่ซ้ำซ้อนกับคำสั่ง
  `saipen validate` + `saipen status`

สิ่งเหล่านี้เคยถูกเสนอและประเมินมาแล้วก่อนหน้านี้ การรื้อฟื้นพวกมัน
จำเป็นต้องใช้หลักฐานใหม่ ไม่ใช่แค่ถามคำถามซ้ำ
